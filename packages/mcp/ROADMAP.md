# ROADMAP

## Стандартизировать entry point для каждого компонента

Сейчас встречаются такие варианты

```json
[
    "packages/*/src/Component.tsx",
    "packages/*/src/Component.ts",
    "packages/*/src/component.tsx",
    "packages/*/src/Component.responsive.tsx",
    "packages/*/src/component.responsive.tsx"
]
```

Также для `icon-view` в принципе нет входной точки в корне пакета

## Проработать генерацию пропсов

Сейчас пропсы берутся из компонента, который находится в entry point пакета. Необходимо брать пропсы всех подкомпонентов, так как это тоже api, и агент должен о них знать

```jsx
<ArgsTabs
    components={{
        'UniversalModal.Header': UniversalModal.Header,
        'UniversalModal.Content': UniversalModal.Content,
        'UniversalModal.Footer': UniversalModal.Footer,
    }}
/>
```

## Обработка deprecate компонентов

На данный момент исключил из генерации компоненты deprecate секции. В следующих итерациях нужно будет решить, что с ними делать - собирать данные по ним или нет

## Описание компонента

Сейчас забираем из children простой регуляркой. Стоит в дальнейшем рассмотреть более детерминированные способы получения description компонента

```mdxjs
<ComponentHeader
    name='Accordion'
    children='Используется для отображения скрытого контента по клику.'
/>
```

## Импорты

В альфа версии для каждого компонента был поле с информацией об импортах

```json
{
    "imports": [
        {
            "from": "@alfalab/core-components/accordion",
            "named": ["Accordion"]
        }
    ]
}
```

Скорее всего будет достаточным держать эту информацию в скилле. Например: описать импорты атомов, импорт из рута, импорты desktop/mobile версий

## Примеры кода

На данный момент самое узкое место по сборку данных. Все демки находтся в docs/desription.mdx. Приходится просто парсить файл и получать данные. Но сами по себе эти данныхе могут быть местами мусорными, и давать агенту не чистые данные. Нужно рассмотреть варианты, как можно было бы получить примеры кода более детерминированно

## Skill

Для более качественно работы необходимо написать скилл. Как работать с импортами, как работать с компонентами, как работать с темизацией и переменными. Обогадить некоторыми данными из текущих страниц сторибука. В общем дать макисмально подробоную инфу по работе с библиотекой, потому что одного mcp будет недостаточно.

## Scenario

Для каждой демки нужен сценарий использонвания (success, error, etc.). По факту сценарии уже могут быть описаны, просто само демо может являться комбайном разных сценаривев. Необходимо делать примеры боллее атомарными
